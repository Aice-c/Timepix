# 重构方向记录

这份文档记录已经和用户确认过的取舍，避免后续 agent 把临时脚本或旧实现当成最终设计。

## 0. 第一阶段重构状态

已新增第一阶段自动化实验系统：

- `timepix/`：新核心包。
- `configs/`：数据集和实验 YAML。
- `scripts/train.py`：单实验入口。
- `scripts/run_grid.py`：网格实验入口。
- `scripts/summarize.py`：结果汇总入口。

旧 `Program/` 暂时保留，不删除。后续应逐步把仍有价值的旧模型、图表生成、报告生成逻辑迁移到新系统。

## 1. `run_ablation.py` 的定位

`Program/run_ablation.py` 只是临时消融实验脚本，主要用于快速对比：

- 不同损失函数。
- 软标签与普通标签。
- 分类任务与回归任务。

它不是论文最终实验框架，也不一定代表最终消融实验的划分方式。后续重构时，可以暂时不优先处理它，而是设计更通用的实验配置与结果记录系统。

## 2. 未来实验框架需要支持的对比维度

用户希望后续不是手工跑零散实验，而是写实验脚本自动执行系列对比并记录结果。可能需要对比：

- 不同模型架构。
- 不同损失函数。
- 不同标签编码方式，例如 one-hot、软标签。
- 分类任务与回归任务。
- 是否启用手工特征。
- 是否启用多模态。
- alpha 与 C/质子数据集上的不同实验设置。

注意：alpha 有 ToT/ToA 双模态，C/质子只有 ToT。

## 3. 模型适配

当前 `Program/model/` 中有很多过时模型。未来需要统一适配项目当前状态，但具体保留哪些架构、每个架构需要支持哪些能力，稍后再讨论。

未来模型统一接口至少应考虑：

- 单模态和多模态输入通道。
- 分类任务。
- 可能的回归任务。
- 可选手工特征输入。
- 是否使用简单拼接分类器。
- 是否使用注意力式特征融合。

## 4. 动态计算输入通道

应使用动态输入通道计算，避免 `config.inchannel` 在 `modalities` 改变后过期。

推荐方向：

- 删除或弱化静态 `config.inchannel`。
- 统一使用 `config.input_channels()` 或等价的配置对象属性。
- 模型构造时从当前实验配置读取输入通道，而不是依赖导入时写死的 class 属性。

## 5. `sweep.py` 的定位

当前 `Program/sweep.py` 不建议只做小修小补。原因：

- 目前 objective 返回值错误，返回了 `run_experiment` 的 dict。
- 搜索空间比较随意，未与实验设计强绑定。
- 结果和参数记录不够全面。
- 不适合支撑论文里的系统性对比实验。

建议后续和新的实验 runner 一起重构，或者删除旧脚本后重写。

## 6. 指标优化原则

可以记录多种指标，但主优化目标暂时以 accuracy 为主更合理，尤其当前任务是离散角度分类。

建议记录：

- validation accuracy。
- test accuracy。
- angle MAE by argmax。
- angle MAE by probability-weighted angle。
- P90 Error。
- confusion matrix。
- per-class precision/recall/F1。

模型选择可以先以 validation accuracy 为主，同时把 MAE 作为重要辅助指标。这样既符合分类任务直觉，也能在论文中体现角度有序性的物理意义。

## 7. 实验元数据是否重要

实验元数据很重要，尤其这个项目会频繁比较模型、损失、标签、模态和手工特征。没有完整元数据，后续很难回答：

- 某个结果到底用的哪个数据集？
- 使用了哪些模态？
- 是否开了手工特征？
- 训练/验证/测试如何划分？
- 模型参数量是多少？
- 损失函数和标签编码是什么？
- 当前结果能不能复现？
- 多个实验之间是否公平可比？

建议每次实验至少保存一个 `metadata.json`，包含：

- 数据集名称和路径。
- 模态列表。
- label map。
- train/val/test split 信息或 split 文件路径/hash。
- 模型名和模型参数量。
- 任务类型。
- 损失函数和标签编码。
- 主要超参数。
- 手工特征配置。
- 标准化配置。
- 最佳 epoch。
- validation/test 指标。
- 输出文件路径。
